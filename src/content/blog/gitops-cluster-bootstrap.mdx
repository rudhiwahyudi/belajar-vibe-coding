Every time our team spun up a new Kubernetes cluster — for a new client, a new environment, or disaster recovery testing — it took roughly **two days** of manual setup: provisioning, installing add-ons, configuring ingress and TLS, wiring up monitoring. This post covers how I turned that into a toolkit that does it in under an hour.

## The problem with "just clone the last cluster's config"

The obvious shortcut — copying YAML from the last cluster — created drift almost immediately. Every cluster ended up slightly different, which made debugging cross-environment issues painful and onboarding new engineers slower than it needed to be.

## The approach: Terraform + ArgoCD app-of-apps

The toolkit has two layers:

### 1. Terraform modules for cluster provisioning

Two modules — one for AWS (EKS) and one for GCP (GKE) — provision the cluster, node pools, and the minimum networking required (VPC, subnets, security groups/firewall rules). Both modules expose the same output interface, so the next layer doesn't need to know which cloud it's running on.

```hcl
module "cluster" {
  source = "./modules/eks" # or ./modules/gke
  name   = "staging"
  region = "ap-southeast-1"
  node_groups = {
    default = { instance_type = "t3.medium", min_size = 2, max_size = 5 }
  }
}
```

### 2. ArgoCD "app of apps"

Once the cluster exists, a single `bootstrap` ArgoCD Application points at a repo containing an `ApplicationSet`. That ApplicationSet generates one Application per add-on:

- `cert-manager` for automatic TLS certificates
- `ingress-nginx` for ingress routing
- `kube-prometheus-stack` for metrics and alerting
- Environment-specific overlays via Kustomize

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: cluster-addons
spec:
  generators:
    - git:
        repoURL: https://github.com/rudhiwahyudi/k8s-gitops-toolkit
        directories:
          - path: addons/*
  template:
    metadata:
      name: '{{path.basename}}'
    spec:
      source:
        repoURL: https://github.com/rudhiwahyudi/k8s-gitops-toolkit
        path: '{{path}}'
      destination:
        server: https://kubernetes.default.svc
```

## Results

After packaging this up:

- New cluster bootstrap time dropped from **~2 days to under 1 hour**
- Every cluster now has the same baseline observability stack, so dashboards and alert rules just work
- Two other teams adopted the toolkit for their own staging environments

## What I'd do differently

If I started over, I'd introduce **policy-as-code (OPA/Gatekeeper)** from day one rather than retrofitting it later — a few of the early clusters needed manual remediation once we added baseline security policies.

The full toolkit is open source: [k8s-gitops-toolkit on GitHub](https://github.com/rudhiwahyudi/k8s-gitops-toolkit).
